模块代码架构说明

本文档面向 HABIT 开发者，用于快速理解各主要模块的代码组织、调用链、 数据契约和扩展点。整体依赖规则与包级拓扑见 HABIT 整体架构（V1）- 整合文档；本文更关注“读代码时从哪里开始、改功能时改哪里”。

总览

HABIT 的主干能力可以按下面的数据链理解：

CLI / Python API
  -> Domain Configurator
  -> Domain Workflow / Pipeline
  -> CSV / NRRD / PKL artifacts
  -> downstream workflow

三个业务域之间不直接 import 彼此的内部实现：

  • habit.core.preprocessing 负责图像预处理，输出标准化影像。

  • habit.core.habitat_analysis 负责 habitat 分割、habitat map 与特征表。

  • habit.core.machine_learning 负责读取 CSV 特征并训练、预测、比较模型。

它们通过文件产物衔接，而不是通过跨域对象引用衔接。跨域共享能力放在 habit.core.common 和 habit.utils。

标准调用链

habit <command> -c config.yaml
  -> habit/cli.py
  -> cmd_<domain>.run_*
  -> ConfigClass.from_file(...)
  -> DomainConfigurator(...)
  -> create_<service>()
  -> service.run / fit / predict

命令组织有两类：

  • 业务流程命令 ：preprocess 、sort-dicom 、get-habitat 、extract 、model 、 cv 、compare 、icc 、radiomics 、retest。这些命令通常 走 YAML 配置和 domain configurator。

  • 辅助工具命令 ：dicom-info 、merge-csv 、dice。它们主要由命令行 参数驱动，dicom-info 和 merge-csv 有对应 cmd_*.py ，dice 直接从 habit/cli.py 调用 habit.utils.dice_calculator。

维护注意事项

  • 新增业务命令时，优先新增 cmd_<name>.py ，再在 habit/cli.py 注册。

  • 新增需要 YAML 的业务服务时，先定义 BaseConfig 子类，再在对应 domain configurator 添加 create_* 方法。

  • configurator 内部应只负责装配，不应承载业务运行逻辑。

habit.core.preprocessing

模块定位

preprocessing 子包负责把原始 DICOM/NIfTI 数据变成后续 habitat 或 radiomics 可直接读取的标准化影像。它是图像层面的批处理 pipeline， 不要和 habitat_analysis 内部的 subject/group feature preprocessing 混淆。

关键文件

文件

职责

config_schemas.py

PreprocessingConfig 和每个步骤配置的 Pydantic schema。

image_processor_pipeline.py

BatchProcessor ，按 subject 并行调度整条预处理链。

base_preprocessor.py

BasePreprocessor ，所有具体预处理器的统一接口。

preprocessor_factory.py

PreprocessorFactory ，通过注册名创建具体 preprocessor。

load_image.py

LoadImagePreprocessor ，把路径读取成 SimpleITK.Image。

dcm2niix_converter.py

Dcm2niixConverter ，调用外部 dcm2niix 执行 DICOM 转换。

resample.py / registration/ / n4_correction.py 等

具体预处理步骤，均通过 factory 注册。

数据流

PreprocessingConfig
  -> PreprocessingConfigurator.create_batch_processor()
  -> BatchProcessor.run()
  -> LoadImagePreprocessor
  -> PreprocessorFactory.create(step_name)
  -> BasePreprocessor.__call__(subject_data)
  -> processed images on disk

BatchProcessor 的核心循环是按 subject 处理。每个 subject 的数据放在 subject_data 字典中，常见键包括：

  • subj ：subject ID。

  • 影像 modality 键，例如 delay2 、t1 、flair。

  • mask 键，例如 mask_delay2 或配置中指定的 mask 名。

  • output_dirs ：当前 subject 的输出目录集合。

配置中的 Preprocessing 字段是有序的「步骤名 -> 参数」映射。步骤名必须 等于 PreprocessorFactory 中注册的名字；每个步骤通常通过 images 字段指定要处理的 modality。

扩展点

新增图像预处理步骤时：

  1. 新建继承 BasePreprocessor 的类。

  2. 实现 __call__(self, data: Dict[str, Any]) -> Dict[str, Any]。

  3. 用 PreprocessorFactory.register("step_name") 注册。

  4. 确保模块会被 habit.core.preprocessing 导入，否则注册装饰器不会执行。

  5. 在配置模板或用户文档中补充该 step_name 的参数说明。

维护注意事项

  • LoadImagePreprocessor 是 BatchProcessor 的隐式前置步骤，通常不需要 用户在 YAML 中写 load_image。

  • 预处理器应尽量只读写 subject_data 中自己声明处理的键，避免隐式修改 其它步骤依赖的数据。

  • 进度条统一使用 habit.utils.progress_utils.CustomTqdm 或经 parallel_utils 间接使用。

habit.core.habitat_analysis

模块定位

habitat_analysis 子包负责从多模态影像和 ROI mask 中生成 habitat map、 habitat 标签表和可复用的训练 pipeline。它也包含 habitat map 后续特征抽取 和传统 radiomics 抽取的实现。

关键文件

文件

职责

habitat_analysis.py

HabitatAnalysis deep module，负责 build、fit、predict、持久化和结果后处理； 内部以 _PIPELINE_RECIPES 字典分发 clustering mode。

config_schemas.py

Pydantic 模型：HabitatAnalysisConfig / FeatureConstructionConfig / HabitatSegmentationConfig；ResultColumns 集中保留列名常量。

HabitatSubjectData ：个体级各 step 传递的显式数据（字段随流水线逐步填充）。

pipelines/steps/*.py

10 个具体步骤（见下表）。supervoxel_aggregation.py 标 DEPRECATED， 仅为向后兼容反序列化保留导入。

services/feature_service.py

特征抽取与 subject 级预处理、supervoxel 文件发现、supervoxel 级特征计算。

services/clustering_service.py

聚类算法创建、最佳聚类数选择、可视化、训练状态字段。

K-Means、GMM、SLIC、Hierarchical 等聚类算法实现，注册装饰器 @register_clustering 暴露给 ClusteringService。

clustering_features/

聚类前的体素 / supervoxel 级特征抽取器与方法表达式解析 （raw 、kinetic 、local_entropy 、mean_voxel_features 、 supervoxel_radiomics 等）。

feature_preprocessing/

生境**特征**预处理（与 habit.core.preprocessing 图像预处理不同）： 统一 DataFrame 进/出 pipeline（pipeline.apply_preprocessing_pipeline）； 内置方法通过 @register_preprocessing 注册到 PreprocessingMethodFactory；群体级状态由 PreprocessingState （baseline + 每步 step_states）维护。

habitat_features/

已生成 habitat map 上的特征抽取与分析（HabitatMapAnalyzer 等）。

Pipeline 结构与执行模型

HabitatAnalysis 通过 _PIPELINE_RECIPES 按 config.HabitatSegmentation.clustering_mode 选择 recipe：

  • two_step ：voxel → supervoxel（个体级）→ 群体级 habitat（跨 subject 聚类）。

  • one_step ：每个 subject 内直接 voxel → habitat。

  • direct_pooling ：跨 subject pooling 后统一群体聚类。

recipe 返回 (name, step) 列表，交给 HabitatPipeline。pipeline 在 __init__ 时按 step 继承的标记类自动分两类，对应两个执行阶段：

  • IndividualLevelStep ：逐 subject 处理。Pipeline 用 parallel_map 把 单个 subject 串行经过所有个体级 step 当作原子单元，按 config.processes 并行多个 subject。这意味着峰值内存大致 processes × 单 subject 内存 ，可由配置精确控制。

  • GroupLevelStep ：所有 subject 汇总后顺序执行；包含训练阶段 有状态 的 GroupPreprocessingStep 与 GroupClusteringStep。

Step 在两阶段之间通过统一的字典契约衔接（见下方 Pipeline 步契约）。

状态管理

判断 step 是否需要保存状态（即是否需要在 fit 时记下、在 transform 时复用）的核心原则：

  • 是否依赖训练数据本身计算的统计量？ 个体级步骤都用每个 subject 自身 的数据计算（特征提取/Z-score/聚类簇数），跨数据集独立 → 无状态。

  • 是否需要跨 subject 共享同一参数？ 群体级预处理（如全局分箱）和群体 级聚类（habitat 模型）必须用训练集统计量与已训练模型，否则训练集和测试 集的 habitat 定义会漂移 → 有状态。

V1 把状态完全放在 step 内部：GroupPreprocessingStep 自带 PreprocessingState ，GroupClusteringStep 自带 clustering_model 与 optimal_n_clusters_。HabitatPipeline.save/load 用 joblib 一并 序列化整条 pipeline——不需要外部 TrainingMode/TestingMode 类。

训练数据流（two-step 完整顺序）

images + masks  (Dict[subject_id, {}])
  -> VoxelFeatureExtractor                  # individual, stateless
  -> IndividualPreprocessingStep            # individual, stateless
  -> IndividualClusteringStep               # individual, stateless
  -> CalculateMeanVoxelFeaturesStep         # individual, stateless
  -> [SupervoxelFeatureExtractionStep]      # individual, stateless, optional
  -> MergeSupervoxelFeaturesStep            # individual, stateless
  -> CombineSupervoxelsStep                 # group, stateless
  -> [GroupPreprocessingStep]               # group, STATEFUL, optional
  -> GroupClusteringStep               # group, STATEFUL
  => habitats.csv + habitat maps + habitat_pipeline.pkl

one-step 在 IndividualClusteringStep 之后只接 CalculateMean… 、 MergeSupervoxelFeatures… 、CombineSupervoxels… ，不进群体级聚类。 direct_pooling 在 IndividualPreprocessing 之后直接 ConcatenateVoxels → 可选 GroupPreprocessing → GroupClustering。

预测数据流

预测 不会 重新 fit pipeline：

  1. HabitatPipeline.load(pipeline_path) 反序列化训练产物，fitted_=True。

  2. HabitatAnalysis._inject_services_into_pipeline 用显式白名单 _PIPELINE_SERVICE_ATTRS 把当前运行时的 service 注入到每个 step。

  3. FeatureService 只同步当前 images_paths 、mask_paths 、日志目标， 保留训练时的 fitted 状态。

  4. pipeline.config.plot_curves 被强制置为 False——预测路径下没有 selection 的中间分数，关闭以保护绘图代码。

  5. 调用 pipeline.transform(X) 生成新数据结果。

并行与内存

  • 个体级阶段由 HabitatPipeline._process_subjects_parallel 并行执行； processes 即并行 subject 数。

  • 每个并行 worker 内 step 顺序执行；step 实现内部 不需要 写并行逻辑。

  • 群体级阶段单进程顺序执行，因为输入是一张大 DataFrame。

  • mask_info 在 worker 内被收集到 pipeline 主进程的 mask_info_cache ， 供 HabitatResultPublisher / HabitatImageWriter 在流水线结束后落盘标签图时复用。

扩展点

新增 habitat 聚类模式时：

  • 新增一个 recipe 函数 _build_<mode>_steps ，返回 `` (name, step)`` 列表。

  • 把模式名加入 _PIPELINE_RECIPES。

  • 输出列差异（如 one-step 把 Supervoxel 镜像为 Habitats）放在对应 step 内；pipeline 之后的 habitats.csv 与可选标签图发布由 HabitatResultPublisher.publish 统一处理。

新增 pipeline step 时：

  • 继承 IndividualLevelStep 或 GroupLevelStep ，明确状态语义。

  • transform 必须保持上下游字典契约的兼容性。

  • 需要共享服务时通过构造函数接收 service，不要在 step 内重新创建。

  • 在本节 Pipeline 步契约 表中追加该 step 的一行。

新增生境特征预处理方法时：

  1. 在 feature_preprocessing/ 下实现 BaseFeaturePreprocessing 子类 （fit / transform ，输入输出均为 feature DataFrame）。

  2. 用 @register_preprocessing("method_name") 注册；若会删列，设 changes_columns = True。

  3. 在 config_schemas.PreprocessingMethod.method 的 Literal 中追加方法名。

  4. 将 changes_columns=True 的方法名同步到 DROPPING_PREPROCESSING_METHODS （与 two_step subject 级校验一致）。

  5. 参考模板 feature_preprocessing/custom_preprocessing_template.py。

  6. 确保模块在运行时被 import 一次，使注册装饰器生效（可参考 builtin_methods.py 的加载方式）。

个体级与群体级共用同一套 handler；差别仅在于群体级通过 PreprocessingState 在训练时缓存 baseline 与各步 step_states ， 预测时复用。

新增 service 时：

  • 在 services/ 下新建 service 类，并在 HabitatAnalysis.__init__ 注入。

  • 必须 把它的属性名加入 _PIPELINE_SERVICE_ATTRS ，否则 predict 路径 不会重新注入它（这是有意的强制约定）。

维护注意事项

  • 旧 strategies/ 子包和 pipeline_builder.py 已删除；不要按旧三层结构 新增代码。

  • service 注入用显式白名单 _PIPELINE_SERVICE_ATTRS；新增 service 必须 主动修改白名单与 _inject_services_into_pipeline 的调用方。

  • 用户向使用与配置说明见 生境分割； 域级架构（recipe / 服务 / 持久化 / 设计决策）见 HABIT 整体架构（V1）- 整合文档 的 ②-a 节。

habit.core.machine_learning

模块定位

machine_learning 子包读取 CSV 特征表，完成训练、预测、K 折验证、模型比较、 统计检验、绘图和报告输出。它不关心 CSV 来自 habitat、传统 radiomics 还是临床表。

训练调用链

MLConfig
  -> MLConfigurator.create_ml_workflow()
  -> HoldoutWorkflow.run()
  -> DataManager
  -> PipelineBuilder
  -> FeatureSelectTransformer
  -> scaler / imputer
  -> ModelFactory.create_model()
  -> reporting components
  -> model PKL + reports + figures

PipelineBuilder 负责把 YAML 配置转为 sklearn Pipeline。常见顺序是：

imputer
  -> feature selection before scaling
  -> scaler
  -> feature selection after scaling
  -> model

FeatureSelectTransformer 在 sklearn pipeline 内调用 run_selector。 selector 注册表会按 selector 函数签名注入 X 、y 、 selected_features 、outdir 等参数。

预测和比较

  • 预测模式仍使用 HoldoutWorkflow （旧名 MachineLearningWorkflow 仍可用），由 MLConfig.run_mode 分发。 预测时加载训练好的 *_final_pipeline.pkl ，输出预测结果和可选评估指标。

  • ModelComparison 不训练模型，只读取多个模型的预测结果，组合 MultifileEvaluator 、Plotter 、ThresholdManager 、ReportExporter 生成比较图、统计检验和报告。

扩展点

新增模型时：

  1. 继承项目模型基类或兼容 sklearn estimator 接口。

  2. 用 ModelFactory.register("model_name") 注册。

  3. 在配置 schema 或模板中补充可用参数。

  4. 若模型需要额外依赖，应保持可选依赖的 import 边界清楚。

新增特征选择方法时：

  1. 在 feature_selectors 中实现 selector 函数。

  2. 用注册机制加入 selector_registry。

  3. 明确 selector 属于 scaling 前还是 scaling 后。

  4. 保持输入输出为特征名列表或可被 FeatureSelectTransformer 消费的结构。

维护注意事项

  • V1 中训练和预测共用 MLConfig 与 HoldoutWorkflow； 不要恢复独立 PredictionWorkflow。

  • 绘图文字必须使用英文。

  • workflow 之外的评估、绘图、报告逻辑尽量放入 evaluation 、 visualization 、reporting 或 callback，保持 workflow 只负责编排。

habit.core.common

模块定位

common 是 core 内的共享基础设施层，主要处理配置、服务装配和少量横切工具。 它可以被业务域使用，但不应在模块顶层 import 任一业务域的重型实现。

关键文件

文件

职责

configs/base.py

BaseConfig 、ConfigAccessor；统一 from_file / from_dict / to_dict。

configs/loader.py

YAML/JSON 读写和路径解析。

configs/validator.py

ConfigValidator 与 load_and_validate_config。

configurators/

域专用服务装配入口。

dataframe/

表格和数组清洗辅助。

边界

  • configs.loader 解决“配置文件如何读、路径如何解析”。

  • BaseConfig 和各 domain schema 解决“配置结构是否合法”。

  • configurators 解决“已验证配置如何组装成可运行服务”。

不要把业务运行逻辑放进 common。如果一个函数需要理解 habitat 或 ML 的领域 语义，它应该留在对应 domain 子包。

habit.utils

模块定位

utils 是全包可用的横切工具层。它不依赖 habit.core ，因此可以被 preprocessing、habitat、ML 和 CLI 辅助工具共同使用。

开发者阅读顺序

建议按下面顺序阅读代码：

  1. 先读 HABIT 整体架构（V1）- 整合文档，掌握全局依赖方向。

  2. 根据要改的功能选择一个 domain：

    • 图像预处理：从 BatchProcessor 和 PreprocessorFactory 开始。

    • habitat 分割：从 HabitatAnalysis 和 _PIPELINE_RECIPES 开始。

    • ML 建模：从 HoldoutWorkflow 、BaseWorkflow 、 PipelineBuilder 开始。

  3. 再深入到具体 step、model、selector、extractor。

  4. 最后查看对应配置 schema 和配置模板，确认用户可配置面。

维护时优先保持“入口薄、domain 内聚、domain 间通过文件契约解耦”的结构。
